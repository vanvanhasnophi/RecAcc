## 饿了么推荐系统业务背景

在饿了么App中，推荐系统主要服务于**首页推荐、类目、搜索**等流量入口，覆盖了90%以上的订单。算法的优化目标包括：

- **点击率**：用户是否点击推荐的餐厅/食物
- **转化率**：用户是否完成下单
- **客单价/单均价**：订单金额
- **用户满意度**：复购、评分等

你的数据应该是用于**排序模型（Ranking Model）**训练的样本，每条记录代表一个**用户-餐厅-上下文**的交互实例。

## 字段业务含义详解

### 一、核心标识字段

| 字段名 | 业务含义 | 说明 |
|:---|:---|:---|
| `label` | 是否下单/点击 | 0=未转化，1=转化（可能是点击或下单） |
| `user_id` | 用户唯一标识 | 哈希处理后的用户ID，用于关联用户画像 |
| `times` | 时间戳 | 行为发生时间，Unix时间戳格式 |
| `geohash12` | 位置编码 | 用户收货地址/当前位置的geohash编码，用于距离计算 |

### 二、用户维度特征

| 字段名 | 业务含义 | 说明 |
|:---|:---|:---|
| `gender` | 用户性别 | 1/2或其他编码 |
| `visit_city` | 访问城市ID | 用户当前所在城市 |
| `avg_price` | 用户历史平均消费 | 用户过去订单的平均金额，反映消费能力 |
| `is_supervip` | 是否为超级会员 | 0=普通用户，1=会员 |
| `ctr_30` | 近30天点击率 | 用户近30天的点击率，反映活跃度 |
| `ord_30` | 近30天下单次数 | 用户近30天的订单数量 |
| `total_amt_30` | 近30天消费总额 | 用户近30天的总消费金额 |

### 三、餐厅/商品维度特征

| 字段名 | 业务含义 | 说明 |
|:---|:---|:---|
| `shop_id` | 餐厅ID | 当前推荐的餐厅 |
| `item_id` | 商品ID | 当前推荐的具体商品 |
| `city_id` | 城市ID | 餐厅所在城市 |
| `district_id` | 区域ID | 餐厅所在行政区 |
| `shop_aoi_id` | 商圈ID | AOI=Area of Interest，商业区/商圈编码 |
| `shop_geohash_6/12` | 餐厅位置编码 | geohash精度6≈1.2km，12≈3.7cm |
| `brand_id` | 品牌ID | 餐厅所属品牌（如肯德基、麦当劳） |
| `category_1_id` | 一级品类ID | 如"快餐简餐"、"奶茶甜品" |
| `merge_standard_food_id` | 标准化菜品ID | 跨餐厅的标准菜品（如"鱼香肉丝"） |

### 四、排名/热度特征

| 字段名 | 业务含义 | 说明 |
|:---|:---|:---|
| `rank_7` | 近7天销量排名 | 在同类目中的销量排名，值越小排名越高 |
| `rank_30` | 近30天销量排名 | 同上 |
| `rank_90` | 近90天销量排名 | 同上 |

**特别注意**：这些是**排名**，值越小代表排名越靠前（如rank_7=1表示近7天销量第1名）

### 五、用户行为序列特征

这些是用户在**当前下单前**的历史行为序列，是推荐系统的核心特征。

| 字段名 | 业务含义 | 序列内容 |
|:---|:---|:---|
| `shop_id_list` | 浏览的餐厅序列 | 用户点击/浏览过的餐厅ID，按时间顺序 |
| `item_id_list` | 浏览的商品序列 | 用户点击/浏览过的商品ID |
| `category_1_id_list` | 浏览的品类序列 | 用户点击过的品类ID |
| `merge_standard_food_id_list` | 浏览的菜品序列 | 用户点击过的标准化菜品ID |
| `brand_id_list` | 浏览的品牌序列 | 用户点击过的品牌ID |
| `price_list` | 浏览商品的价格序列 | **数值序列**，用户看到的商品价格 |
| `shop_aoi_id_list` | 浏览餐厅的商圈序列 | 用户点击过的餐厅所在商圈 |
| `shop_geohash6_list` | 浏览餐厅的位置序列 | 用户点击过的餐厅位置 |
| `timediff_list` | **时间差序列** | **关键特征**：每次行为距离当前的时间差（秒/分钟） |
| `hours_list` | 行为发生的小时序列 | 每次行为发生在几点（0-23） |
| `time_type_list` | 行为发生的餐别序列 | breakfast/lunch/dinner/night/tea |
| `weekdays_list` | 行为发生的星期序列 | 0-6代表周一到周日 |

#### `timediff_list`的特殊业务含义

这个字段是我们之前讨论的**有序数值**的典型案例，在饿了么推荐系统中有重要含义：

- **用户实时兴趣衰减**：时间差越小，代表行为越近期，对当前决策影响越大
- **会话（Session）边界**：时间差突增可能表示用户开启了新会话
- **从你的第二行数据看**：前17个值（47-557）可能是**秒级**时间差（几分钟内），后4个（25389-25602）可能是**分钟级**（约4-7小时），表示用户可能分两次访问

| 字段名 | 业务含义 | 说明 |
|:---|:---|:---|
| `hours` | 当前小时 | 下单/请求发生时的小时 |
| `time_type` | 当前餐别 | breakfast/lunch/dinner/night/tea |
| `weekdays` | 当前星期 | 0-6 |

## 字段关联的业务逻辑

### 推荐模型的输入逻辑

在饿了么的排序模型中，通常会融合多维度特征：

1. **用户画像特征**：gender, avg_price, is_supervip, ctr_30, ord_30, total_amt_30
   - 反映用户长期偏好和消费能力

2. **餐厅/商品画像特征**：brand_id, category_1_id, shop_aoi_id
   - 反映餐厅的属性

3. **上下文特征**：hours, time_type, weekdays
   - 反映当前场景（午饭时间还是夜宵）

4. **用户实时行为序列**：shop_id_list, timediff_list, price_list等
   - 反映用户**本次会话**中的实时兴趣
   - 通过时间差衰减计算用户兴趣强度

5. **交叉特征**：用户-餐厅距离、用户历史对该餐厅的点击率等

### 为什么`timediff_list`如此重要？

在饿了么的在线学习系统中，**实时反馈特征**是关键优化点：

> "随后基于业务的理解将其拆分为点击率、转化率2个子模型，并引入用户、商户的实时反馈特征，如用户点击某个餐厅、餐厅近一个小时或者一天的情况，效果提升7%-8%。"

`timediff_list`正是这种**实时反馈特征**的体现——记录用户本次会话中每次行为的**时间衰减因子**。

## 总结：各字段的业务价值

| 字段类别 | 业务价值 | 建模中的作用 |
|:---|:---|:---|
| 核心ID | 实体关联 | 拼接画像、做embedding |
| 用户画像 | 长期偏好 | 理解用户消费能力、活跃度 |
| 餐厅画像 | 物品属性 | 理解餐厅品类、品牌、位置 |
| 排名 | 物品热度 | 反映物品的流行度 |
| 行为序列 | 实时兴趣 | **核心特征**，反映用户当前意图 |
| 上下文 | 场景感知 | 理解时间、餐别对决策的影响 |

**特别说明`timediff_list`**：它代表了**用户兴趣的时间衰减权重**，在点击率(CTR)预估模型中通常与行为序列配合使用——越近期的行为权重越高，越能反映用户当前的即时需求。
